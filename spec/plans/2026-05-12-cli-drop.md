# cli/drop — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the `ingitdb drop` command — positional `<kind> <name>` invocation supporting `collection` and `view` kinds, `--if-exists` for idempotency, `--cascade` flag (no-op in the current data model — see notes), local source. Removes both the schema-entry side (the line in `.ingitdb/root-collections.yaml` for collections, or the view-definition file for views) and the data side (the collection's data directory, or the view's materialized output file) in a single operation.

**Architecture:** A new file `cmd/ingitdb/commands/drop.go` hosts the `Drop` command. cobra positional subcommands handle the kind discriminator (`drop collection <name>` and `drop view <name>` map to dedicated subcommand RunE handlers). For `drop collection`, the implementation reads `.ingitdb/root-collections.yaml`, looks up the collection's relative directory, removes the entire subtree (which transitively removes views nested under `$views/`), then rewrites the root-collections file with the entry removed. For `drop view`, the implementation scans each collection's `$views/` directory for a matching view name, removes the view-definition file, and removes the view's materialized output file (per `ViewDef.FileName`).

**Tech Stack:** Go stdlib (`os`, `path/filepath`), `gopkg.in/yaml.v3`, `github.com/spf13/cobra`, the project's `pkg/ingitdb` types.

**Spec:** `spec/features/cli/drop/README.md`

## Data-model reality check

The cli/drop spec was approved assuming a dependency graph between collections and views. The actual ingitdb data model is simpler:

- **Views are children of their parent collection** — stored on disk under `<collection-dir>/$views/<view-name>.yaml`, not at the top level. A view cannot reference a collection from "outside" because it IS inside a collection's directory tree.
- **No explicit cross-object foreign keys** — there is no declared dependency from one collection to another.

What this means for `--cascade`:

- `drop collection <name>` removes the entire collection directory subtree. Views nested under `$views/` go away automatically. There are no "external" dependents to cascade to. **`--cascade` is implicitly always-on for collections, so the flag is structurally a no-op.**
- `drop view <name>` has no dependents in this data model. The flag is a no-op for views as well.

What this means for `req:dependents-error-by-default`:

- There are no detectable cross-object dependencies, so the "block on dependents" branch can never trigger. The REQ is preserved in the spec as a forward-compatible contract; the plan does NOT implement dependency detection. This is documented in the spec-coverage section of the self-review.

The plan registers `--cascade` and accepts it without error (no-op), but does NOT block invocations without it. If a future data-model change introduces cross-object references, dependency detection can be added without breaking the existing flag surface.

```go
package commands

import (
	"fmt"
	"os"
	"path/filepath"

	"gopkg.in/yaml.v3"
)

// rootCollectionsRelPath is the on-disk location of the root
// collections registry, relative to the database root.
const rootCollectionsRelPath = ".ingitdb/root-collections.yaml"

// readRootCollections reads the database's root-collections.yaml file
// and returns it as a map of collection-name → relative directory
// path.
func readRootCollections(dbDir string) (map[string]string, error) {
	path := filepath.Join(dbDir, rootCollectionsRelPath)
	raw, err := os.ReadFile(path)
	if err != nil {
		return nil, fmt.Errorf("read %s: %w", rootCollectionsRelPath, err)
	}
	entries := make(map[string]string)
	if unmarshalErr := yaml.Unmarshal(raw, &entries); unmarshalErr != nil {
		return nil, fmt.Errorf("parse %s: %w", rootCollectionsRelPath, unmarshalErr)
	}
	return entries, nil
}

// writeRootCollectionsWithout reads the root-collections.yaml file,
// removes the entry for the named collection (no-op if absent), and
// writes the file back. Other entries are preserved with their
// original values.
func writeRootCollectionsWithout(dbDir, dropName string) error {
	entries, err := readRootCollections(dbDir)
	if err != nil {
		return err
	}
	delete(entries, dropName)
	out, err := yaml.Marshal(entries)
	if err != nil {
		return fmt.Errorf("marshal root-collections: %w", err)
	}
	path := filepath.Join(dbDir, rootCollectionsRelPath)
	if writeErr := os.WriteFile(path, out, 0o644); writeErr != nil {
		return fmt.Errorf("write %s: %w", rootCollectionsRelPath, writeErr)
	}
	return nil
}
```

## Task 3 — `drop collection <name>` implementation

**Context:** `drop collection cities` does three things in order: (1) read root-collections.yaml to discover the collection's on-disk directory, (2) remove that directory subtree (which transitively removes views nested under `$views/`), (3) rewrite root-collections.yaml without the entry. The order matters: if the file rewrite happens first and the directory remove fails, the on-disk state diverges from the schema. Doing the directory remove first means a half-completed drop leaves a deleted directory with a still-listed entry, which the user can recover by manually deleting the entry. The reverse failure mode is less recoverable.

Per the spec's `req:drop-collection-removes-schema-and-data`, both must succeed for a "drop". We implement step (2) then step (3), with a wrapper that reports a clear diagnostic naming the failed stage.
