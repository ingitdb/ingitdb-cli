Documentation inGitDB

inGitDB transforme un référentiel Git en une base de données validée par schéma, interrogeable, où chaque enregistrement est un simple fichier YAML ou JSON. Il n'y a pas de serveur à exécuter et pas de format binaire à décoder - les données se trouvent dans des fichiers que vous pouvez lire, éditer et comparer avec n'importe quel éditeur de texte ou client git.

Pourquoi inGitDB?

Git lui-même fournit l'historique, la branchement et l'historique de fusion gratuitement. Si cela semble être le bon compromis pour votre projet, lisez la suite et envisagez de contribuer.

Démarrage rapide

Commencez avec inGitDB en quelques commandes seulement:

Créer le CLI

go build -o ingitdb ./cmd/ingitdb

Valider un répertoire de base de données

# Valider le répertoire actuel
ingitdb validate

# Valider un chemin spécifique
ingitdb validate --path=/path/to/your/db

# Mode CI rapide - uniquement les enregistrements modifiés
ingitdb validate --from-commit=abc1234 --to-commit=def5678

Lister les collections

# Lister toutes les collections
ingitdb list collections

# Lister les collections imbriquées dans un chemin
ingitdb list collections --in='countries/(ie|gb)'

# Filtrer par nom
ingitdb list collections --filter-name='*city*'

Opérations CRUD

# Créer un enregistrement
ingitdb create record --id=geo/nations/ie --data='{title: "Ireland"}'

# Lire un enregistrement
ingitdb read record --id=geo/nations/ie

# Mettre à jour un enregistrement
ingitdb update record --id=geo/nations/ie --set='{capital: "Dublin"}'

# Supprimer un enregistrement
ingitdb delete record --id=geo/nations/ie

Commandes CLI

Référence complète de toutes les commandes disponibles:

Commande Statut Description
version Implémenté Afficher la version de la compilation, le hash du commit et la date
validate Implémenté Vérifier chaque enregistrement par rapport au schéma de sa collection
read record Implémenté Lire un seul enregistrement par ID (local ou GitHub)
create record Implémenté Créer un nouvel enregistrement (local ou GitHub)
update record Implémenté Mettre à jour les champs d'un enregistrement existant
delete record Implémenté Supprimer un seul enregistrement par ID
list collections Implémenté Lister les ID de collections (local ou GitHub)
find Prévu Rechercher les enregistrements par sous-chaîne, regex ou valeur exacte
query Prévu Interroger et formater les enregistrements d'une collection
materialize Prévu Construire les vues matérialisées dans $views/
watch Prévu Diffuser les événements de modification d'enregistrement vers stdout
serve Prévu Démarrer le serveur MCP, HTTP API ou file-watcher
Format --id

Le drapeau --id utilise / comme séparateur: <collection-path>/<record-key>

Exemple: --id=geo/nations/ie

Configuration

Un fichier .ingitdb.yaml minimal à la racine de votre référentiel de base de données:

rootCollections:
  tasks: data/tasks/*   # chaque sous-répertoire devient une collection
languages:
  - required: en

Apprenez-en plus sur les options de configuration dans la documentation de configuration.

Accès direct à GitHub

inGitDB peut lire et écrire des enregistrements dans un référentiel GitHub distant sans le cloner. Le drapeau --github remplace --path et pointe le CLI vers un référentiel GitHub via l'API REST.

Référentiels publics (aucun jeton requis)

# Lire un enregistrement
ingitdb read record --github=owner/repo --id=countries/ie

# Épingler à une branche, un tag ou un SHA de commit spécifique
ingitdb read record --github=owner/repo@main --id=todo.tags/active
ingitdb read record --github=owner/repo@v1.2.0 --id=todo.tags/active

# Lister toutes les collections
ingitdb list collections --github=owner/repo

Référentiels privés

# Définir le jeton une fois dans votre shell
export GITHUB_TOKEN=ghp_...

# Maintenant vous pouvez lire et écrire
ingitdb read record --github=owner/repo --id=countries/ie
ingitdb create record --github=owner/repo --id=countries/ie --data='{name: Ireland}'
ingitdb update record --github=owner/repo --id=countries/ie --set='{capital: Dublin}'
ingitdb delete record --github=owner/repo --id=countries/ie

Chaque opération d'écriture crée un seul commit dans le référentiel distant. Aucun clone local requis.

Architecture

Le diagramme ci-dessous montre le chemin d'écriture: une commande CLI charge la définition de la base de données, valide les enregistrements, reconstruit les vues matérialisées et laisse chaque changement comme un commit Git normal.

Fichiers YAML/JSON → Référentiel Git → Validation → Générateur de vues → Sortie (CLI/Go/IA)

Décisions clés de conception

  • Fichiers ordinaires: Chaque enregistrement est un fichier YAML ou JSON que vous pouvez éditer dans n'importe quel éditeur de texte
  • Git comme stockage: Tous les contrôles de version, les branches et l'historique proviennent de Git
  • Validation du schéma: Les définitions de colonnes typées garantissent l'intégrité des données
  • Aucun serveur pour la lecture: La lecture des données n'est que des opérations du système de fichiers
  • Interfaces standard: Intégration DALgo pour Go, MCP pour agents IA

En savoir plus dans la documentation sur l'architecture.

Bibliothèque Go (DALgo)

pkg/dalgo2ingitdb implémente l'interface DALgo dal.DB, donc n'importe quel programme Go peut utiliser inGitDB comme une abstraction de base de données standard.

import "github.com/ingitdb/ingitdb-cli/pkg/dalgo2ingitdb"

// Utiliser inGitDB comme instance dal.DB
db := dalgo2ingitdb.NewDatabase(path)

// Opérations DALgo standard
// ... votre code ici ...

Cela vous permet de basculer entre différents backends de bases de données (Firestore, SQL, inGitDB) sans modifier votre code d'application.

Contribuer

inGitDB est assez petit pour que chaque contribution fasse une différence visible. La meilleure façon de commencer est de pointer le CLI sur un répertoire de fichiers YAML et d'exécuter ingitdb validate.

Comment contribuer

  1. Lire CONTRIBUTING.md pour le flux de travail de pull-request
  2. Parcourir les problèmes ouverts pour trouver quelque chose sur lequel travailler
  3. Ouvrir ou commenter un problème avant d'investir du temps dans un grand changement

Les rapports de bugs, les améliorations de documentation et les questions sont bienvenues.

Feuille de route

inGitDB est développé en neuf phases de livraison:

Phase 1: Validateur de schéma ✅

Validation du schéma, CRUD basique, listage des collections

Phase 2: Moteur de requête ⏳

Recherche, filtrage et opérations de requête

Phase 3: Constructeur de vues ⏳

Vues matérialisées et agrégations

Phase 4: Flux d'événements ⏳

Watcher de fichiers et flux d'événements de modification

Phase 5: API HTTP ⏳

Serveur API REST pour accès distant

Phase 6: Serveur MCP ⏳

Model Context Protocol pour agents IA

Phases 7-9: Avenir

Outils de migration, déclencheurs et API GraphQL
