Документація inGitDB

inGitDB перетворює репозиторій Git на валідовану за схемою, запитувану базу даних, де кожен запис - це простий файл YAML або JSON. Немає сервера для запуску та немає двійкового формату для декодування - дані знаходяться у файлах, які ви можете читати, редагувати та порівнювати за допомогою будь-якого текстового редактора або клієнта git.

Чому inGitDB?

Git сам по собі надає історію, розгалуження та історію об'єднання безкоштовно. Якщо це звучить як правильний компроміс для вашого проекту, читайте далі та розглядайте можливість сприяння.

Швидкий старт

Почніть роботу з inGitDB всього кількома командами:

Побудова CLI

go build -o ingitdb ./cmd/ingitdb

Валідуйте директорію бази даних

# Валідуйте поточну директорію
ingitdb validate

# Валідуйте конкретний шлях
ingitdb validate --path=/path/to/your/db

# Швидкий режим CI - тільки змінені записи
ingitdb validate --from-commit=abc1234 --to-commit=def5678

Список колекцій

# Список усіх колекцій
ingitdb list collections

# Список колекцій вкладених у шлях
ingitdb list collections --in='countries/(ie|gb)'

# Фільтр за іменем
ingitdb list collections --filter-name='*city*'

Операції CRUD

# Створіть запис
ingitdb create record --id=geo/nations/ie --data='{title: "Ireland"}'

# Прочитайте запис
ingitdb read record --id=geo/nations/ie

# Оновіть запис
ingitdb update record --id=geo/nations/ie --set='{capital: "Dublin"}'

# Видаліть запис
ingitdb delete record --id=geo/nations/ie

Команди CLI

Повний довідник усіх доступних команд:

Команда Статус Опис
version Реалізовано Виведіть версію побудови, хеш коміту та дату
validate Реалізовано Перевірте кожен запис у схемі його колекції
read record Реалізовано Прочитайте один запис за ID (локально або на GitHub)
create record Реалізовано Створіть новий запис (локально або на GitHub)
update record Реалізовано Оновіть поля існуючого запису
delete record Реалізовано Видаліть один запис за ID
list collections Реалізовано Список ID колекцій (локально або на GitHub)
find Планується Пошук записів за підстроку, регулярним виразом або точним значенням
query Планується Запитайте та відформатуйте записи з колекції
materialize Планується Побудуйте матеріалізовані подання у $views/
watch Планується Потокова передача подій зміни запису на stdout
serve Планується Запустіть сервер MCP, HTTP API або спостереження файлів
Формат --id

Прапор --id використовує / як розділювач: <collection-path>/<record-key>

Приклад: --id=geo/nations/ie

Конфігурація

Мінімальний .ingitdb.yaml у корені вашого репозиторію бази даних:

rootCollections:
  tasks: data/tasks/*   # кожна підпапка стає колекцією
languages:
  - required: en

Дізнайтеся більше про параметри конфігурації в документації з конфігурації.

Прямий доступ до GitHub

inGitDB може читати та писати записи у віддаленому репозиторії GitHub без його клонування. Прапор --github замінює --path та спрямовує CLI на репозиторій GitHub через REST API.

Публічні репозиторії (токен не потрібен)

# Прочитайте запис
ingitdb read record --github=owner/repo --id=countries/ie

# Закріпіть на конкретній гілці, теґу або хеші коміту
ingitdb read record --github=owner/repo@main --id=todo.tags/active
ingitdb read record --github=owner/repo@v1.2.0 --id=todo.tags/active

# Список усіх колекцій
ingitdb list collections --github=owner/repo

Приватні репозиторії

# Встановіть токен один раз у вашій оболонці
export GITHUB_TOKEN=ghp_...

# Тепер ви можете читати та писати
ingitdb read record --github=owner/repo --id=countries/ie
ingitdb create record --github=owner/repo --id=countries/ie --data='{name: Ireland}'
ingitdb update record --github=owner/repo --id=countries/ie --set='{capital: Dublin}'
ingitdb delete record --github=owner/repo --id=countries/ie

Кожна операція запису створює один коміт у віддаленому репозиторії. Локальний клон не потрібен.

Функції

Що inGitDB може робити сьогодні

  • ✅ Валідація схеми з типізованими визначеннями стовпців
  • ✅ CRUD операції на записах (колекції map[string]any)
  • ✅ Прямий доступ до репозиторію GitHub (читання та запис)
  • ✅ Список колекцій з фільтрами регулярних виразів
  • ✅ Швидка інкрементна валідація (діапазон коміту)
  • ✅ Відстеження версій з історією git

Що наступає

  • ⏳ Повнотекстовий пошук з підтримкою регулярних виразів
  • ⏳ Матеріалізовані подання
  • ⏳ Сервер MCP для інтеграції агента ШІ
  • ⏳ Спостереження файлів для подій зміни в реальному часі
  • ⏳ HTTP API сервер
  • ⏳ Інтерактивне розв'язання конфліктів об'єднання

Архітектура

Діаграма нижче показує шлях запису: команда CLI завантажує визначення бази даних, валідує записи, перебудовує матеріалізовані подання та залишає кожну зміну як звичайний git коміт.

Файли YAML/JSON → Репозиторій Git → Валідація → Конструктор представлень → Вихід (CLI/Go/ШІ)

Ключові рішення про дизайн

  • Звичайні файли: Кожен запис - це файл YAML або JSON, який ви можете редагувати в будь-якому текстовому редакторі
  • Git як сховище: Весь контроль версій, розгалуження та історія походять з Git
  • Валідація схеми: Типізовані визначення стовпців забезпечують цілісність даних
  • Нема сервера для читання: Читання даних - це просто операції файлової системи
  • Стандартні інтерфейси: Інтеграція DALgo для Go, MCP для агентів ШІ

Дізнайтеся більше в документації з архітектури.

Бібліотека Go (DALgo)

pkg/dalgo2ingitdb реалізує інтерфейс DALgo dal.DB, тому будь-яка програма на Go може використовувати inGitDB як абстракцію стандартної бази даних.

import "github.com/ingitdb/ingitdb-cli/pkg/dalgo2ingitdb"

// Використовуйте inGitDB як екземпляр dal.DB
db := dalgo2ingitdb.NewDatabase(path)

// Стандартні операції DALgo
// ... ваш код тут ...

Це дозволяє вам перейти між різними серверними базами даних (Firestore, SQL, inGitDB) без змін коду вашої програми.

Сприяння

inGitDB досить малий, щоб кожен внесок мав видимий вплив. Найкращий спосіб почати - це спрямувати CLI на папку файлів YAML та запустити ingitdb validate.

Як сприяти

  1. Прочитайте CONTRIBUTING.md для рабочого процесу pull-request
  2. Перегляньте відкриті проблеми, щоб знайти що-небудь для роботи
  3. Відкрийте або прокоментуйте проблему перед інвестуванням часу у велику зміну

Звіти про помилки, вдосконалення документації та питання приветствуються.

Дорожна карта

inGitDB розробляється в дев'яти фазах доставки:

Фаза 1: Валідатор схеми ✅

Валідація схеми, базовий CRUD, список колекцій

Фаза 2: Механізм запитів ⏳

Пошук, фільтрація та операції запиту

Фаза 3: Конструктор представлень ⏳

Матеріалізовані подання та агрегації

Фаза 4: Потокова передача подій ⏳

Спостереження файлів та потоки подій змін

Фаза 5: HTTP API ⏳

REST API сервер для віддаленого доступу

Фаза 6: Сервер MCP ⏳

Model Context Protocol для агентів ШІ

Фази 7-9: Майбутнє

Інструменти міграції, тригери та GraphQL API
