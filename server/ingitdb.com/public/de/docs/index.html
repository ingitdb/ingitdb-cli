inGitDB-Dokumentation

inGitDB verwandelt ein Git-Repository in eine schemavalidierte, abfragbare Datenbank, in der jeder Eintrag eine normale YAML- oder JSON-Datei ist. Es gibt keinen Server zum Ausführen und kein Binärformat zum Dekodieren - die Daten befinden sich in Dateien, die Sie mit jedem Texteditor oder Git-Client lesen, bearbeiten und vergleichen können.

Warum inGitDB?

Git selbst bietet Verlauf, Verzweigung und Zusammenführungsverlauf kostenlos. Wenn das wie der richtige Kompromiss für Ihr Projekt klingt, lesen Sie weiter und erwägen Sie, beizutragen.

Schnelleinstieg

Beginnen Sie mit inGitDB mit nur wenigen Befehlen:

CLI erstellen

go build -o ingitdb ./cmd/ingitdb

Datenbankverzeichnis validieren

# Validieren Sie das aktuelle Verzeichnis
ingitdb validate

# Validieren Sie einen bestimmten Pfad
ingitdb validate --path=/path/to/your/db

# Schneller CI-Modus - nur geänderte Einträge
ingitdb validate --from-commit=abc1234 --to-commit=def5678

Sammlungen auflisten

# Alle Sammlungen auflisten
ingitdb list collections

# Sammlungen in einem Pfad auflisten
ingitdb list collections --in='countries/(ie|gb)'

# Nach Name filtern
ingitdb list collections --filter-name='*city*'

CRUD-Operationen

# Eintrag erstellen
ingitdb create record --id=geo/nations/ie --data='{title: "Ireland"}'

# Eintrag lesen
ingitdb read record --id=geo/nations/ie

# Eintrag aktualisieren
ingitdb update record --id=geo/nations/ie --set='{capital: "Dublin"}'

# Eintrag löschen
ingitdb delete record --id=geo/nations/ie

CLI-Befehle

Vollständige Referenz aller verfügbaren Befehle:

Befehl Status Beschreibung
version Implementiert Buildversion, Commit-Hash und Datum ausgeben
validate Implementiert Jeden Eintrag gegen sein Sammlungsschema überprüfen
read record Implementiert Einen einzelnen Eintrag nach ID lesen (lokal oder auf GitHub)
create record Implementiert Neuen Eintrag erstellen (lokal oder auf GitHub)
update record Implementiert Felder eines vorhandenen Eintrags aktualisieren
delete record Implementiert Einen einzelnen Eintrag nach ID löschen
list collections Implementiert Sammlungs-IDs auflisten (lokal oder auf GitHub)
find Geplant Einträge nach Teilstring, Regex oder genauem Wert durchsuchen
query Geplant Einträge aus einer Sammlung abfragen und formatieren
materialize Geplant Materialisierte Ansichten in $views/ erstellen
watch Geplant Eintragänderungsereignisse an stdout streamen
serve Geplant MCP-, HTTP-API- oder Datei-Watcher-Server starten
--id-Format

Das Flag --id verwendet / als Trennzeichen: <collection-path>/<record-key>

Beispiel: --id=geo/nations/ie

Konfiguration

Minimale .ingitdb.yaml im Stamm Ihres Datenbankrepositories:

rootCollections:
  tasks: data/tasks/*   # jedes Unterverzeichnis wird eine Sammlung
languages:
  - required: en

Erfahren Sie mehr über Konfigurationsoptionen in der Konfigurationsdokumentation.

Direkter GitHub-Zugriff

inGitDB kann Einträge in einem Remote-GitHub-Repository lesen und schreiben, ohne es zu klonen. Das Flag --github ersetzt --path und verweist die CLI über die REST API auf ein GitHub-Repository.

Öffentliche Repositorys (kein Token erforderlich)

# Eintrag lesen
ingitdb read record --github=owner/repo --id=countries/ie

# An einen bestimmten Branch, Tag oder Commit-SHA anheften
ingitdb read record --github=owner/repo@main --id=todo.tags/active
ingitdb read record --github=owner/repo@v1.2.0 --id=todo.tags/active

# Alle Sammlungen auflisten
ingitdb list collections --github=owner/repo

Private Repositorys

# Setzen Sie das Token einmal in Ihrer Shell
export GITHUB_TOKEN=ghp_...

# Jetzt können Sie lesen und schreiben
ingitdb read record --github=owner/repo --id=countries/ie
ingitdb create record --github=owner/repo --id=countries/ie --data='{name: Ireland}'
ingitdb update record --github=owner/repo --id=countries/ie --set='{capital: Dublin}'
ingitdb delete record --github=owner/repo --id=countries/ie

Jeder Schreibvorgang erstellt einen einzelnen Commit im Remote-Repository. Kein lokaler Klon erforderlich.

Funktionen

Was inGitDB heute tun kann

  • ✅ Schemavalidierung mit typisierten Spaltendefinitionen
  • ✅ CRUD-Operationen auf Einträgen (map[string]any-Sammlungen)
  • ✅ Direkter GitHub-Repository-Zugriff (lesen und schreiben)
  • ✅ Sammlungsauflistung mit Regex-Filtern
  • ✅ Schnelle inkrementelle Validierung (Commit-Bereich-Scoped)
  • ✅ Versionsverfolgung mit Git-Verlauf

Was kommt

  • ⏳ Volltextsuche mit Regex-Unterstützung
  • ⏳ Materialisierte Ansichten
  • ⏳ MCP-Server für KI-Agent-Integration
  • ⏳ Datei-Watcher für Echtzeit-Änderungsereignisse
  • ⏳ HTTP-API-Server
  • ⏳ Interaktive Konfliktauflösung beim Zusammenführen

Architektur

Das folgende Diagramm zeigt den Schreibpfad: Ein CLI-Befehl lädt die Datenbankdefinition, validiert Einträge, erstellt materialisierte Ansichten neu und hinterlässt jede Änderung als normalen Git-Commit.

YAML/JSON-Dateien → Git-Repository → Validierung → Ansicht-Builder → Ausgabe (CLI/Go/KI)

Wichtige Designentscheidungen

  • Normale Dateien: Jeder Eintrag ist eine YAML- oder JSON-Datei, die Sie in jedem Texteditor bearbeiten können
  • Git als Speicher: Alle Versionskontrolle, Verzweigung und Verlauf stammen aus Git
  • Schemavalidierung: Typisierte Spaltendefinitionen gewährleisten Datenintegrität
  • Kein Server zum Lesen: Das Lesen von Daten ist nur eine Dateisystem-Operation
  • Standard-Schnittstellen: DALgo-Integration für Go, MCP für KI-Agenten

Erfahren Sie mehr in der Architekturdokumentation.

Go-Bibliothek (DALgo)

pkg/dalgo2ingitdb implementiert die DALgo dal.DB Schnittstelle, daher kann jedes Go-Programm inGitDB als Standard-Datenbankabstraktion verwenden.

import "github.com/ingitdb/ingitdb-cli/pkg/dalgo2ingitdb"

// Verwenden Sie inGitDB als dal.DB-Instanz
db := dalgo2ingitdb.NewDatabase(path)

// Standard-DALgo-Operationen
// ... Ihr Code hier ...

Dies ermöglicht es Ihnen, zwischen verschiedenen Datenbankbackends (Firestore, SQL, inGitDB) zu wechseln, ohne Ihren Anwendungscode zu ändern.

Beitragen

inGitDB ist klein genug, dass jeder Beitrag einen sichtbaren Unterschied macht. Der beste Weg zu beginnen ist, die CLI auf ein Verzeichnis mit YAML-Dateien zu verweisen und ingitdb validate auszuführen.

Wie man beiträgt

  1. Lesen Sie CONTRIBUTING.md für den Pull-Request-Workflow
  2. Durchsuchen Sie offene Probleme, um etwas zum Arbeiten zu finden
  3. Öffnen oder kommentieren Sie ein Problem, bevor Sie Zeit in eine große Änderung investieren

Fehlermeldungen, Dokumentationsverbesserungen und Fragen sind willkommen.

Roadmap

inGitDB wird in neun Lieferphasen entwickelt:

Phase 1: Schema Validator ✅

Schemavalidierung, grundlegender CRUD, Sammlungsauflistung

Phase 2: Query Engine ⏳

Suche, Filterung und Abfrageoperationen

Phase 3: Views Builder ⏳

Materialisierte Ansichten und Aggregationen

Phase 4: Event Streaming ⏳

Datei-Watcher und Änderungsereignisströme

Phase 5: HTTP API ⏳

REST-API-Server für Remote-Zugriff

Phase 6: MCP Server ⏳

Model Context Protocol für KI-Agenten

Phasen 7-9: Zukunft

Migrations-Tools, Trigger und GraphQL API
