Documentación de inGitDB

inGitDB convierte un repositorio Git en una base de datos validada por esquema y consultable, donde cada registro es un archivo YAML o JSON simple. No hay servidor que ejecutar ni formato binario que decodificar: los datos residen en archivos que puede leer, editar y comparar con cualquier editor de texto o cliente git.

¿Por qué inGitDB?

Git en sí proporciona historial, ramificación e historial de fusión gratis. Si eso suena como el equilibrio correcto para su proyecto, siga leyendo y considere contribuir.

Inicio rápido

Comience con inGitDB con solo unos pocos comandos:

Construir CLI

go build -o ingitdb ./cmd/ingitdb

Validar un directorio de base de datos

# Validar el directorio actual
ingitdb validate

# Validar una ruta específica
ingitdb validate --path=/path/to/your/db

# Modo rápido CI - solo registros modificados
ingitdb validate --from-commit=abc1234 --to-commit=def5678

Listar colecciones

# Listar todas las colecciones
ingitdb list collections

# Listar colecciones anidadas en una ruta
ingitdb list collections --in='countries/(ie|gb)'

# Filtrar por nombre
ingitdb list collections --filter-name='*city*'

Operaciones CRUD

# Crear un registro
ingitdb create record --id=geo/nations/ie --data='{title: "Ireland"}'

# Leer un registro
ingitdb read record --id=geo/nations/ie

# Actualizar un registro
ingitdb update record --id=geo/nations/ie --set='{capital: "Dublin"}'

# Eliminar un registro
ingitdb delete record --id=geo/nations/ie

Comandos CLI

Referencia completa de todos los comandos disponibles:

Comando Estado Descripción
version Implementado Imprimir versión de compilación, hash de commit y fecha
validate Implementado Verificar cada registro contra su esquema de colección
read record Implementado Leer un único registro por ID (local o GitHub)
create record Implementado Crear un nuevo registro (local o GitHub)
update record Implementado Actualizar campos de un registro existente
delete record Implementado Eliminar un único registro por ID
list collections Implementado Listar ID de colecciones (local o GitHub)
find Planeado Buscar registros por subcadena, regex o valor exacto
query Planeado Consultar y formatear registros de una colección
materialize Planeado Construir vistas materializadas en $views/
watch Planeado Transmitir eventos de cambio de registro a stdout
serve Planeado Iniciar servidor MCP, HTTP API o file-watcher
Formato --id

La bandera --id usa / como separador: <collection-path>/<record-key>

Ejemplo: --id=geo/nations/ie

Configuración

Un archivo .ingitdb.yaml mínimo en la raíz de su repositorio de base de datos:

rootCollections:
  tasks: data/tasks/*   # cada subdirectorio se convierte en una colección
languages:
  - required: en

Obtenga más información sobre las opciones de configuración en la documentación de configuración.

Acceso directo a GitHub

inGitDB puede leer y escribir registros en un repositorio remoto de GitHub sin clonarlo. La bandera --github reemplaza --path y apunta la CLI a un repositorio de GitHub a través de la API REST.

Repositorios públicos (no se requiere token)

# Leer un registro
ingitdb read record --github=owner/repo --id=countries/ie

# Fijar en una rama, etiqueta o SHA de commit específicos
ingitdb read record --github=owner/repo@main --id=todo.tags/active
ingitdb read record --github=owner/repo@v1.2.0 --id=todo.tags/active

# Listar todas las colecciones
ingitdb list collections --github=owner/repo

Repositorios privados

# Establecer el token una vez en su shell
export GITHUB_TOKEN=ghp_...

# Ahora puede leer y escribir
ingitdb read record --github=owner/repo --id=countries/ie
ingitdb create record --github=owner/repo --id=countries/ie --data='{name: Ireland}'
ingitdb update record --github=owner/repo --id=countries/ie --set='{capital: Dublin}'
ingitdb delete record --github=owner/repo --id=countries/ie

Cada operación de escritura crea un solo commit en el repositorio remoto. No se requiere clon local.

Características

Lo que inGitDB puede hacer hoy

  • ✅ Validación de esquema con definiciones de columnas tipadas
  • ✅ Operaciones CRUD en registros (colecciones map[string]any)
  • ✅ Acceso directo al repositorio de GitHub (lectura y escritura)
  • ✅ Listado de colecciones con filtros regex
  • ✅ Validación incremental rápida (rango de commit)
  • ✅ Seguimiento de versiones con historial git

Lo que viene

  • ⏳ Búsqueda de texto completo con soporte regex
  • ⏳ Vistas materializadas
  • ⏳ Servidor MCP para integración de agentes de IA
  • ⏳ Vigilante de archivos para eventos de cambio en tiempo real
  • ⏳ Servidor API HTTP
  • ⏳ Resolución interactiva de conflictos de fusión

Arquitectura

El diagrama a continuación muestra la ruta de escritura: un comando CLI carga la definición de la base de datos, valida registros, reconstruye vistas materializadas y deja cada cambio como un commit git normal.

Archivos YAML/JSON → Repositorio Git → Validación → Generador de vistas → Salida (CLI/Go/IA)

Decisiones clave de diseño

  • Archivos ordinarios: Cada registro es un archivo YAML o JSON que puede editar en cualquier editor de texto
  • Git como almacenamiento: Todo el control de versiones, ramificación e historial proviene de Git
  • Validación de esquema: Las definiciones de columnas tipadas garantizan la integridad de los datos
  • Sin servidor para lectura: Leer datos es solo operaciones del sistema de archivos
  • Interfaces estándar: Integración de DALgo para Go, MCP para agentes de IA

Obtenga más información en la documentación de arquitectura.

Biblioteca Go (DALgo)

pkg/dalgo2ingitdb implementa la interfaz DALgo dal.DB, por lo que cualquier programa Go puede usar inGitDB como una abstracción de base de datos estándar.

import "github.com/ingitdb/ingitdb-cli/pkg/dalgo2ingitdb"

// Usar inGitDB como una instancia dal.DB
db := dalgo2ingitdb.NewDatabase(path)

// Operaciones DALgo estándar
// ... su código aquí ...

Esto le permite cambiar entre diferentes backends de bases de datos (Firestore, SQL, inGitDB) sin cambiar el código de su aplicación.

Contribuir

inGitDB es lo suficientemente pequeño para que cada contribución marque una diferencia visible. La mejor manera de comenzar es apuntar la CLI a un directorio de archivos YAML y ejecutar ingitdb validate.

Cómo contribuir

  1. Lea CONTRIBUTING.md para el flujo de trabajo de solicitud de extracción
  2. Examine los problemas abiertos para encontrar algo en lo que trabajar
  3. Abra o comente un problema antes de invertir tiempo en un cambio importante

Los informes de errores, las mejoras de documentación y las preguntas son bienvenidas.

Hoja de ruta

inGitDB se desarrolla en nueve fases de entrega:

Fase 1: Validador de esquema ✅

Validación de esquema, CRUD básico, listado de colecciones

Fase 2: Motor de consulta ⏳

Búsqueda, filtrado y operaciones de consulta

Fase 3: Constructor de vistas ⏳

Vistas materializadas y agregaciones

Fase 4: Transmisión de eventos ⏳

Vigilante de archivos y flujos de eventos de cambio

Fase 5: API HTTP ⏳

Servidor API REST para acceso remoto

Fase 6: Servidor MCP ⏳

Protocolo de contexto de modelo para agentes de IA

Fases 7-9: Futuro

Herramientas de migración, activadores y API de GraphQL
