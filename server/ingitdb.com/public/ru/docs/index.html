Документация inGitDB

inGitDB превращает репозиторий Git в валидированную по схеме, запрашиваемую базу данных, где каждая запись - это простой файл YAML или JSON. Нет сервера для запуска и никакого двоичного формата для декодирования - данные находятся в файлах, которые вы можете читать, редактировать и сравнивать с помощью любого текстового редактора или клиента git.

Почему inGitDB?

Git сам по себе предоставляет историю, ветвление и историю слияния бесплатно. Если это звучит как правильный компромисс для вашего проекта, читайте дальше и подумайте о способствовании.

Быстрый старт

Начните работу с inGitDB всего с несколько командами:

Сборка CLI

go build -o ingitdb ./cmd/ingitdb

Валидируйте директорию базы данных

# Валидируйте текущую директорию
ingitdb validate

# Валидируйте определённый путь
ingitdb validate --path=/path/to/your/db

# Быстрый режим CI - только изменённые записи
ingitdb validate --from-commit=abc1234 --to-commit=def5678

Список коллекций

# Список всех коллекций
ingitdb list collections

# Список коллекций вложенных в путь
ingitdb list collections --in='countries/(ie|gb)'

# Фильтр по имени
ingitdb list collections --filter-name='*city*'

CRUD операции

# Создайте запись
ingitdb create record --id=geo/nations/ie --data='{title: "Ireland"}'

# Прочитайте запись
ingitdb read record --id=geo/nations/ie

# Обновите запись
ingitdb update record --id=geo/nations/ie --set='{capital: "Dublin"}'

# Удалите запись
ingitdb delete record --id=geo/nations/ie

Команды CLI

Полный справочник всех доступных команд:

Команда Статус Описание
version Реализовано Выведите версию сборки, хэш коммита и дату
validate Реализовано Проверьте каждую запись по схеме её коллекции
read record Реализовано Прочитайте одну запись по ID (локально или на GitHub)
create record Реализовано Создайте новую запись (локально или на GitHub)
update record Реализовано Обновите поля существующей записи
delete record Реализовано Удалите одну запись по ID
list collections Реализовано Список ID коллекций (локально или на GitHub)
find Планируется Поиск записей по подстроке, регулярному выражению или точному значению
query Планируется Запросите и форматируйте записи из коллекции
materialize Планируется Постройте материализованные представления в $views/
watch Планируется Потоковая передача событий изменения записей на stdout
serve Планируется Запустите MCP, HTTP API или сервер наблюдения файлов
Формат --id

Флаг --id использует / в качестве разделителя: <collection-path>/<record-key>

Пример: --id=geo/nations/ie

Конфигурация

Минимальный .ingitdb.yaml в корне вашего репозитория базы данных:

rootCollections:
  tasks: data/tasks/*   # каждая подпапка становится коллекцией
languages:
  - required: en

Узнайте больше о параметрах конфигурации в документации по конфигурации.

Прямой доступ к GitHub

inGitDB может читать и записывать записи в удалённом репозитории GitHub без его клонирования. Флаг --github заменяет --path и направляет CLI на репозиторий GitHub через REST API.

Публичные репозитории (токен не требуется)

# Прочитайте запись
ingitdb read record --github=owner/repo --id=countries/ie

# Закрепите на определённой ветке, теге или хэше коммита
ingitdb read record --github=owner/repo@main --id=todo.tags/active
ingitdb read record --github=owner/repo@v1.2.0 --id=todo.tags/active

# Список всех коллекций
ingitdb list collections --github=owner/repo

Приватные репозитории

# Установите токен один раз в вашей оболочке
export GITHUB_TOKEN=ghp_...

# Теперь вы можете читать и писать
ingitdb read record --github=owner/repo --id=countries/ie
ingitdb create record --github=owner/repo --id=countries/ie --data='{name: Ireland}'
ingitdb update record --github=owner/repo --id=countries/ie --set='{capital: Dublin}'
ingitdb delete record --github=owner/repo --id=countries/ie

Каждая операция записи создаёт один коммит в удалённом репозитории. Локальный клон не требуется.

Возможности

Что inGitDB может делать сегодня

  • ✅ Валидация схемы с типизированными определениями столбцов
  • ✅ CRUD операции на записях (коллекции map[string]any)
  • ✅ Прямой доступ к репозиторию GitHub (чтение и запись)
  • ✅ Список коллекций с фильтрами регулярного выражения
  • ✅ Быстрая пошаговая валидация (с объёмом диапазона коммитов)
  • ✅ Отслеживание версий с историей git

Что грядёт

  • ⏳ Полнотекстовый поиск с поддержкой регулярных выражений
  • ⏳ Материализованные представления
  • ⏳ Сервер MCP для интеграции агента ИИ
  • ⏳ Наблюдатель файлов для событий изменения в реальном времени
  • ⏳ HTTP API сервер
  • ⏳ Интерактивное разрешение конфликтов слияния

Архитектура

Диаграмма ниже показывает путь записи: команда CLI загружает определение базы данных, валидирует записи, перестраивает материализованные представления и оставляет каждое изменение как обычный git коммит.

Файлы YAML/JSON → Репозиторий Git → Валидация → Построитель представлений → Выход (CLI/Go/ИИ)

Ключевые решения по дизайну

  • Обычные файлы: Каждая запись - это файл YAML или JSON, который вы можете редактировать в любом текстовом редакторе
  • Git как хранилище: Весь контроль версий, ветвление и история идут из Git
  • Валидация схемы: Типизированные определения столбцов обеспечивают целостность данных
  • Нет сервера для чтения: Чтение данных - это просто операции файловой системы
  • Стандартные интерфейсы: Интеграция DALgo для Go, MCP для агентов ИИ

Узнайте больше в документации по архитектуре.

Библиотека Go (DALgo)

pkg/dalgo2ingitdb реализует интерфейс DALgo dal.DB, поэтому любая программа на Go может использовать inGitDB как абстракцию стандартной базы данных.

import "github.com/ingitdb/ingitdb-cli/pkg/dalgo2ingitdb"

// Используйте inGitDB как экземпляр dal.DB
db := dalgo2ingitdb.NewDatabase(path)

// Стандартные операции DALgo
// ... ваш код здесь ...

Это позволяет вам переключаться между различными бэкендами базы данных (Firestore, SQL, inGitDB) без изменения кода вашего приложения.

Способствование

inGitDB достаточно мал, чтобы каждый вклад имел видимое значение. Лучший способ начать - это направить CLI на папку файлов YAML и запустить ingitdb validate.

Как способствовать

  1. Прочитайте CONTRIBUTING.md для рабочего процесса pull-request
  2. Просмотрите открытые проблемы, чтобы найти что-то для работы

Отчёты об ошибках, улучшения документации и вопросы приветствуются.

Фаза 1: Валидатор схемы ✅

Триггеры и подписчики

inGitDB может автоматически выполнять пользовательские рабочие процессы при создании, обновлении или удалении записей. Существуют два взаимодополняющих механизма:

Триггеры

Триггеры используют синтаксис, вдохновлённый GitHub Actions. Блок on объявляет, какие события запускают триггер; карта jobs определяет shell-команды для выполнения. Файлы триггеров находятся в .ingitdb-collection/trigger_<name>.yaml.

.ingitdb-collection/trigger_rebuild.yaml

on:
  - created
  - updated

jobs:
  build:
    runs-on: "."
    steps:
      - run: echo "Запись изменена"
      - run: npm run generate

Подписчики

Подписчики — встроенные интеграции, работающие внутри CLI-процесса ingitdb — никаких дополнительных инструментов не требуется. Настраиваются в .ingitdb/subscribers.yaml.

Поле subscribers — это словарь с уникальными идентификаторами, что позволяет адресовать и изменять каждую группу независимо. Каждая группа содержит селектор for — paths (поддерживает литеральные имена, подстановочный знак * и regex-сегменты) и events — а также один или несколько списков обработчиков. Значение представляет собой определение подписчика (SubscriberDef). Под одним селектором можно объединять несколько обработчиков одного или разных типов:

.ingitdb/subscribers.yaml

subscribers:

  notify-backend:
    name: "Уведомить при любом изменении"
    for:
      events:
        - created
        - updated
        - deleted
    webhooks:
      - url: https://api.example.com/hook
        headers:
          Authorization: "Bearer <TOKEN>"

  content-updates:
    name: "Slack, поиск и email при изменении контента"
    for:
      paths:
        - content/posts
        - content/pages
      events:
        - created
        - updated
    slacks:
      - webhook_url: https://hooks.slack.com/services/...
    search_indexes:
      - provider: algolia
        app_id: XXXXXXXXXX
        api_key: "<API_KEY>"
        index: content
    emails:
      - to: [editors@example.com]
        smtp: smtp.example.com
        subject: "{event} on {path}"
Обработчик Статус Описание
webhooks Planned HTTP POST по любому URL
emails Planned SMTP email-уведомление
telegrams Planned Сообщение через Telegram Bot API
slacks Proposal Публикация в канал Slack через входящий вебхук
discords Proposal Публикация в канал Discord через вебхук
whatsapp Proposal Отправка сообщения WhatsApp
github_actions Proposal Запуск события workflow_dispatch
gitlab_ci Proposal Запуск GitLab pipeline
ntfy Proposal Push-уведомления через ntfy.sh (self-hosted)
sms Proposal Отправка SMS через Twilio или Vonage
search_indexes Proposal Синхронизация с Algolia / Meilisearch / Typesense
rss Proposal Генерация файла RSS или Atom feed
